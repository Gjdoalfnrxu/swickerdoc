slots, part 1 of 2

SLOTS!! the WebGL cabinet!!

okay this is the biggest folder and i am SO sorry, look. /frontend/src/slots/ has its OWN sub-architecture:

slots/ ├── SlotsApp.svelte ← top-level. hosts the Threlte canvas. ├── main.ts ← entry, mounts SlotsApp ├── webgl/ ← shaders, reel meshes, particle FX ├── chrome/ ← DOM HUD: balance, bet, spin, auto ├── components/ ← win overlays, leaderboard, bonus screens ├── state/ ← reelStore, winStore, cinematicStore... ├── spinController.ts ← THE conductor. orchestrates everything. ├── timing.ts ← anticipation windows, dwell, frame budgets ├── audioBindings.ts ← stingers + tension layer └── pkg/slots.js ← WASM!! rng + paytable lives in RUST.

the spin pipeline goes: handleSpin → wasm RNG → reelStore.start → useTask raf loop → anticipation if 3+ scatters → resolve → winStore.show → audio stinger. every step is its own module. i broke it apart because the imperative-canvas version was 600 lines of vibes and i couldn't test it.

a basidiocarp drops spores on a frame-perfect cycle — useTask is the same idea. each tick the gills shed one more frame of multiplier into the air.

roulette

roulette: also calm. suspiciously calm.

/frontend/src/roulette/ is just game.ts, sounds.ts, and a __tests__ folder. THAT'S IT. no svelte component yet — the wheel rendering is borrowed from /frontend/src/renderer/roulette/ in the shared mycelium.

fun fact about this: this is actually a really common pattern (split the rules engine from the renderer so the rules can run headless in tests) but i think MY implementation is a bit unhinged because the rules engine is just game.ts exporting pure functions and a TS discriminated union for outcomes. no class, no factory, no DI container. just functions. JUST. FUNCTIONS.

yes i know this isn't industry standard but LISTEN — when the only state is "the wheel landed on 17" you do not need a service locator.

CRASH — the new hotness

CRASH!! the pig climbs the rope ladder!!

OK OK OK OK. /frontend/src/crash/. this is the satire game. a pig with little angel wings climbs a rope ladder, the multiplier ticks up, you cash out before it falls. it is also provably fair via commit-reveal because if we're going to mock the genre we should at least out-engineer it.

// crash/engine/wasmClient.ts
//   1. centralises the WASM import (mockable in vitest)
//   2. unwraps { result } | { error } into a Result<T>
//      discriminated union — no implicit unwrap, no silent swallowing
//   3. coerces performance.now() to bigint for the engine

the rust crate at crates/crash/ owns: the seed schedule, the crash-point distribution, the multiplier curve, the commit hash, the reveal verification. the TS side owns: pixels, taps, vibes. clean seam. you can't cheat the engine from devtools. i tried.

the WASM module is a sclerotium — a dense little knot of dormant compute that wakes up the second nutrients (a player click) hit it, fruits a round, then goes back to sleep. cleanest boundary in the whole repo.

CRASH — the 3D bit

the pig billboard + the InstancedMesh ladder

scene/PigBillboard.svelte renders pig-angel-wings.png on a camera-facing quad. NO SKELETON YET. the climb is just pigY = f(multiplier) with sinusoidal bob + tilt for fake life. phase 3.5 will do the real glTF rig. shipping the billboard first was a deliberate choice — see, i learned about scope from getting burned on slots.

look at the little flourish where the bob frequency drops as the multiplier climbs — pig is getting tired!! i giggled when i wrote it!!

scene/RopeLadder.svelte is ~400 thin Box rungs in a single InstancedMesh from y=0 to y=200, plus two CylinderGeometry side ropes. the whole Group sways ~0.05wu at 0.5Hz for "wind in the rigging" — suppressed under prefers-reduced-motion because we are NICE here.

an instanced mesh is a cluster — one genome, many fruiting bodies, one draw call. that's Armillaria energy. (the world's largest organism is a honey fungus in oregon. unrelated. wanted you to know.)

CRASH — camera follow

the camera follow was a BUG that became a FEATURE.

scene/CrashScene.svelte instantiates the PerspectiveCamera as a sibling of the renderers and lerps camera.position.y toward cameraTargetY at rate 0.1 per tick. originally i forgot to clamp it and the camera overshot the pig on every crash, which looked AMAZING — the pig vanishes upward, the camera lags, you see only sky and falling rope. so i kept it and called it "the abandonment frame".

const CAMERA_LERP_RATE = 0.1;
// Hold the camera at last Y for ~600ms after a crash so the pig is seen
// (or rather, conspicuously isn't). the abandonment frame.

the svelte store pattern

crash/state/crashStore.ts   slots/state/reelStore.ts   blackjack/store.ts

NO wasm calls live in the store. controllers (roundController, spinController) own the engine plumbing and push state-shaped updates in. the store exposes only the snapshot the chrome cares about. one direction, no loops, easy to test.

stores are the mycelial network and components are the fruiting bodies above ground. when the substrate (state) shifts, every mushroom in the patch fruits in sync. that's reactivity, baby.

tests + CI

the spore-print key.

vitest on the TS side, cargo test + proptests on the rust side. the proptests are the part i'm most smug about — they generate random seeds and assert that commit then reveal always verifies, that the multiplier curve is monotonic during Climbing, and that no codepath returns a negative balance.

CI: github actions running on a self-hosted NAS box at home (labels: pesos-deploy, e2e). playwright e2e suite drives the lobby + each game in headless chromium. i did not set up the runners, my devops friend did, but i USE them and i RESPECT them.

a test suite is the mushroom hunter's spore-print key — you can think a thing is Agaricus all day, but you don't know until you put it gill-side-down on white paper overnight. proptests are spore prints.
