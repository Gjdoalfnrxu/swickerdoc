What is Datalog?

Datalog is a tiny logic language. You describe the world with relations and ask questions.

  • Facts — base truths you know directly. parent(tom, bob).
  • Rules — if/then derivations. If X is parent of Y, X is ancestor of Y.
  • Query — what do you want? Who is an ancestor of carol?

No functions. No mutation. No side effects. Everything is a relation, and the engine figures out the rest.

Think of it as SQL's recursive, stateless cousin — tuned for pattern-matching over knowledge graphs.
02 / 08

Facts and Rules — a worked example

We start with base facts about a family:

parent(tom,   bob).
parent(bob,   carol).
parent(carol, dave).

Then we define ancestor with two rules — one base case, one recursive:

ancestor(X, Y) :- parent(X, Y).
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).
The :- reads as "if". Commas mean and. Variables are uppercase, constants lowercase.
03 / 08

Fixpoint Computation

Bottom-up evaluation: start with base facts, apply every rule, repeat until nothing new appears.

RoundNew ancestor facts derived
0(base facts only)
1tom→bob, bob→carol, carol→dave
2tom→carol, bob→dave
3tom→dave
4— nothing new. Done.

The engine stops when a round adds zero new tuples — that's the fixpoint.

Every fact derivable from the rules is now in the database. Guaranteed to terminate for pure Datalog.
04 / 08

The Naive Problem

Fixpoint is elegant. It also computes everything.

  • 50,000 JSX elements in a codebase.
  • You want the 3 that are React Context Providers.
  • Naive evaluation derives every ancestor relationship for every element, then filters.

Per round, per rule: O(n) at best. A badly-ordered join — scanning a big relation before a small one — goes O(n²) or worse. Intermediate state explodes.

Symptom you'll see in the wild: cap-hit. The engine gave up before reaching fixpoint because intermediate relation sizes exceeded budget.
05 / 08

Magic Sets — the key insight

Don't compute all ancestors. Compute only the ancestors of the thing you asked about.

Magic Sets injects a demand predicate that seeds evaluation from the query goal and propagates downward.

% original query
?- ancestor(tom, Y).

% magic seed injected
magic_ancestor(tom).

Every rule body now carries the magic filter. Tuples that don't match the demand are never derived.

Bottom-up evaluation with top-down focus. Keeps Datalog's termination guarantees; adds demand-driven pruning.
06 / 08

How Magic Sets Work

Each rule splits into two: one that propagates demand, one that derives tuples — gated by the demand.

% original
ancestor(X,Y) :- parent(X,Z), ancestor(Z,Y).

% after magic rewrite
magic_ancestor(Z) :- magic_ancestor(X), parent(X,Z).
ancestor(X,Y)    :- magic_ancestor(X), parent(X,Z), ancestor(Z,Y).
  • Rule 1 seeds magic_ancestor for every reachable frontier.
  • Rule 2 only fires when the frontier says "I care about X."
  • Nodes outside the reachable set are never touched.
Orders-of-magnitude reduction in intermediate state is common on big relations with narrow query focus.
07 / 08

When Magic Sets Break

Two well-known failure modes:

  • Disjunction split. A or B in a rule body gets split into two rules. The planner may propagate demand through only one branch — the other scans the full relation unfiltered.
  • Wrong join order. The magic anchor is bound, but the planner starts from the wrong literal. It scans a huge relation first, derives millions of junk tuples, then applies the filter. Same blow-up as naive.
The tell: specific query, magic sets enabled, planner still cap-hits. Check the join order and disjunction shape before blaming the engine.
08 / 08

Best Practices & Limitations

Do:

  • Put magic-anchor-bound variables first in the rule body.
  • Use string constants as filters — a known attribute name beats a universally-quantified variable every time.
  • Avoid deep disjunction chains. Flatten; factor shared subgoals into named predicates.

Don't expect it to save you from:

  • Negation-as-failure inside recursive rules — stratification forbids it.
  • Highly-connected graphs. MayResolveTo on a monorepo touches nearly everything; demand doesn't prune what's genuinely reachable.
Rule of thumb: magic sets pay off when demand is narrow relative to the universe. If you want a lot of the graph, you're going to pay for a lot of the graph.
