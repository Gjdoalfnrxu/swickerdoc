What is Datalog?

Datalog is a tiny logic language. You describe the world with relations and ask questions.

  • Facts — base truths you know directly. parent(tom, bob).
  • Rules — if/then derivations. If X is parent of Y, X is ancestor of Y.
  • Query — what do you want? Who is an ancestor of carol?

No functions. No mutation. No side effects. Everything is a relation, and the engine figures out the rest.

Think of it as SQL's recursive, stateless cousin — tuned for pattern-matching over knowledge graphs.
Slide 2 of 8

Facts and Rules — a worked example

We start with four base facts about a family:

parent(tom,   bob).
parent(bob,   carol).
parent(carol, dave).

Then we define ancestor with two rules — one base case, one recursive:

ancestor(X, Y) :- parent(X, Y).
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).
The :- reads as "if". Commas mean and. Variables are uppercase, constants lowercase.
Slide 3 of 8

Fixpoint Computation

Bottom-up evaluation: start with base facts, apply every rule, repeat until nothing new appears.

RoundNew ancestor facts
0(base facts only)
1tom→bob, bob→carol, carol→dave
2tom→carol, bob→dave
3tom→dave
4— nothing new. Done.

Three rounds closed the chain. The engine stops when a round adds zero new tuples — that's the fixpoint.

Every fact derivable from the rules is now in the database. Guaranteed to terminate for pure Datalog.
Slide 4 of 8

The Naive Problem

Fixpoint is elegant. It also computes everything.

  • 50,000 JSX elements in a codebase.
  • You want the 3 that are React Context Providers.
  • Naive evaluation derives every ancestor relationship for every element, then filters.

Per round, per rule: O(n) at best. A badly-ordered join — scanning a big relation before a small one — goes O(n²) or worse. Intermediate state explodes. The planner hits its cap. The fixpoint blows up.

Symptom you'll see in the wild: cap-hit. The engine gave up before reaching fixpoint because the intermediate relation sizes exceeded its budget.
Slide 5 of 8

Magic Sets — the key insight

Don't compute all ancestors. Compute only the ancestors of the thing you asked about.

Magic Sets is a source-to-source rewrite that injects a demand predicate — a "magic" relation — that seeds evaluation from the query goal and propagates downward.

% original query
?- ancestor(tom, Y).

% magic seed
magic_ancestor(tom).

Now every rule body carries the magic filter. Tuples that don't match the demand are never derived.

Result: bottom-up evaluation with top-down focus. Keeps Datalog's termination guarantees; adds demand-driven pruning.
Slide 6 of 8

How Magic Sets Work

Each original rule becomes two rules: one that propagates demand, one that derives tuples — gated by the demand.

% original recursive rule
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).

% after magic rewrite
magic_ancestor(Z) :- magic_ancestor(X), parent(X, Z).
ancestor(X, Y) :- magic_ancestor(X), parent(X, Z), ancestor(Z, Y).
  • The first new rule seeds magic_ancestor for every reachable recursion frontier.
  • The second only fires when the frontier says "yes, I care about X."
  • Nodes outside the reachable set are never touched.
Dramatic reduction in intermediate state is common — orders of magnitude on big relations with small query focus.
Slide 7 of 8

When Magic Sets Break

Magic Sets is not magic-magic. Two well-known failure modes:

  • Disjunction split. A or B in a rule body gets split into two rules. The planner propagates demand through each independently — and may reach only one branch. The other branch scans the full relation unfiltered.
  • Wrong join order. The magic anchor is bound, but the planner starts evaluation from the wrong literal. It scans a huge relation first, derives millions of junk tuples, then applies the magic filter. Same blow-up as naive.
The tell: you wrote a specific query, magic sets are enabled, the planner still cap-hits. Check the join order and the shape of your disjunctions before blaming the engine.
Slide 8 of 8

Best Practices & Limitations

Do:

  • Put magic-anchor-bound variables first in the rule body. The planner follows positional hints more than it should.
  • Use string constants as filters. They prune dramatically — a known JSX attribute name beats a universally-quantified variable every time.
  • Avoid deep disjunction chains. Flatten where you can; factor shared subgoals.

Don't expect it to save you from:

  • Negation-as-failure inside recursive rules — stratification forbids it, and magic doesn't fix the structure.
  • Highly-connected graphs. MayResolveTo on a monorepo touches nearly everything; demand doesn't prune what's genuinely reachable.
  • Cap-hits as a general symptom — the fixpoint blew up. Magic is a reduction technique, not a size guarantee.
Rule of thumb: magic sets pay off when the demand is narrow relative to the universe. If you want a lot of the graph, you're going to pay for a lot of the graph.
